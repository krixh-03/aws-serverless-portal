DISCREPANCIES BETWEEN IEEE RESEARCH PAPER AND ACTUAL CODEBASE
================================================================

Document: 23EG505R51_CSE_Katamoni Abhishek Goud_UG_Project Report_v1_16-03-2026 new.pdf
Format:   IEEE conference/journal paper (6 pages)

Below are the sections in the IEEE research paper that do not match the
actual implementation in the serverless-portal codebase. Each entry lists
the section, what the paper currently says, and what it should say.


---------------------------------------------------------------------------
1. Section IV (Serverless Computing Architecture) — CloudFront & S3
---------------------------------------------------------------------------
CURRENT (inaccurate):
  "Amazon S3 is used to host static web resources, including HTML, CSS
  as well as JavaScript files. Amazon CloudFront is a content delivery
  network (CDN) that is incorporated to enhance worldwide performance."

CHANGE:
  These services are not deployed in the actual system. The frontend is
  a React + TypeScript SPA served locally via the Vite development server
  (port 5173). Either:
  (a) Reword to state that S3 and CloudFront are part of the designed
      architecture but were not deployed in the current implementation,
      and that the frontend is served locally during development/testing.
  (b) Or actually deploy them by adding S3 bucket and CloudFront
      distribution resources to infrastructure/template.yaml.

2. Section V (System Architecture) — CloudFront & S3 in architecture diagram
---------------------------------------------------------------------------
CURRENT (inaccurate):
  Figure 1 shows the request flow as:
    User → CloudFront CDN → Amazon S3 / API Gateway → Lambda → DynamoDB

  Figure 2 shows:
    User Request → CloudFront CDN → API Gateway → Lambda → DynamoDB

  The accompanying text states: "CloudFront caches the non-living web
  content at distributed points along the edge" and "Amazon S3 is used
  to store such static web assets like HTML, CSS, and JavaScript files."

CHANGE:
  The actual architecture is:
    User → React SPA (Vite localhost:5173) → HTTP API Gateway → Lambda → DynamoDB

  request flow: the React frontend sends API requests directly to the
  HTTP API Gateway endpoint.

3. Section VI (Implementation) — REST API vs HTTP API
---------------------------------------------------------------------------
CURRENT (inaccurate):
  "Amazon API Gateway was configured to expose RESTful API endpoints."

CHANGE:
  The system uses Amazon HTTP API Gateway (v2), not REST API Gateway.
  These are different AWS products. HTTP API is 71% cheaper ($1.00 vs
  $3.50 per million requests) and has lower latency (~5ms vs ~10-15ms
  overhead). Table III at the end of the paper correctly says "HTTP API
  Gateway", so the text in Section VI should match.

  Reword to:
  "Amazon HTTP API Gateway was configured to expose API endpoints.
  HTTP API was chosen over REST API for its significantly lower cost
  and reduced latency overhead."

7. Section VII (Security Considerations) — Authentication mechanism
---------------------------------------------------------------------------
CURRENT (incomplete):
  The section discusses IAM roles, API Gateway throttling, DynamoDB
  encryption, and CloudWatch monitoring. However, it does not mention
  the actual user authentication mechanism used in the system.

CHANGE:
  Add a paragraph describing the actual authentication implementation:
  "User authentication is implemented using BCrypt password hashing
  (cost factor 10) for secure credential storage and JWT (JSON Web
  Token) generation for session management. When a user logs in, the
  auth-login Lambda function queries DynamoDB via a GSI for the user's
  email, verifies the password using BCrypt, and returns a signed JWT
  token. This custom authentication approach was chosen over AWS
  Cognito to minimize cost and maintain simplicity within the AWS
  Free Tier."

  Table III correctly lists "BCrypt + JWT" as the authentication method,
  so the security section should describe this in detail.

8. Section VIII (Performance Evaluation) — No actual numbers in text
---------------------------------------------------------------------------
CURRENT (theoretical only):
  Section VIII discusses cold starts and warm invocations conceptually
  but never references the actual benchmark data from Tables I and II.
  The section reads like a literature review rather than a results
  section.

CHANGE:
  Incorporate the measured results into the text. For example:

  "As shown in Table I, the measured cold start latency was 6.95
  seconds for the first invocation, which includes container
  provisioning (~2.5s), JVM startup (~1.5s), static initialization
  of the DynamoDB client and JWT configuration (918ms), and the
  first request execution including BCrypt verification (~500ms).

  Subsequent warm invocations (Table II) averaged 467ms request
  execution time and 1.30 seconds end-to-end latency, representing
  an 89% reduction from the cold start request time of 4,248ms.
  The residual 467ms is dominated by BCrypt password verification
  (cost factor 10), which is an intentional security measure rather
  than infrastructure overhead. The actual Lambda platform overhead
  on warm invocations is less than 30ms."

  Also reference Table III in Section V or VI where the system
  configuration is discussed.
